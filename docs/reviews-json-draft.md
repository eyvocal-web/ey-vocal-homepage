# EY보컬스튜디오 reviews.json 초안

대상 프로젝트: `ey-vocal-homepage`

목적:
- `#testimonials` placeholder 4건을 이후 실제 후기 데이터로 바꿀 수 있도록 최소 JSON 구조를 먼저 맞춥니다.
- 현재 단계에서는 HTML 교체나 주입 스크립트 추가 없이 스키마와 운영 흐름만 정리합니다.

## 1. 현재 권장 스키마

파일 경로:
- `data/reviews.json`

필드:

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| `id` | string | 후기 고유 ID |
| `nameMask` | string | 공개용 익명 이름 (`김○○`) |
| `date` | string | 게시 월 (`YYYY-MM`) |
| `body` | string | 카드 본문 전체 문장 |
| `source` | string | 후기 출처 (`naver-place`, `manual`, `placeholder`) |
| `rating` | number \| null | 별점 사용 시만 채움 |

익명화 규칙:
- 실명 전체 공개 금지
- `성 + ○○` 형식 유지
- 필요 시 내부 원본과 공개용 값을 분리 관리

현재 드래프트 메모:
- `data/reviews.json`에는 placeholder 4건만 넣었습니다.
- 실제 네이버플레이스 후기로 교체할 때 `source`를 `naver-place`로 바꾸고, `date`와 `body`를 실데이터 기준으로 교체하면 됩니다.

## 2. 주입 방식 비교

### 1순위: 빌드타임 치환

방법:
- 배포 전 스크립트가 `data/reviews.json`을 읽고 `index.html`의 후기 영역을 치환합니다.

장점:
- 정적 사이트 구조와 가장 잘 맞습니다.
- 런타임 요청이 없어 SEO와 초기 렌더링이 안정적입니다.
- 네트워크 실패로 후기 영역이 비어 보일 가능성이 없습니다.

주의:
- 배포 전에 JSON 검수와 치환 스크립트 실행이 필요합니다.

### 2순위: 런타임 fetch

방법:
- 브라우저에서 `data/reviews.json`을 fetch해서 후기 DOM을 교체합니다.

장점:
- HTML 템플릿과 데이터 파일을 느슨하게 분리할 수 있습니다.

단점:
- 자바스크립트 의존도가 늘어납니다.
- 초기 SEO/SSR 이점이 줄고, 로딩 타이밍 관리가 추가됩니다.
- 현재 순수 정적 구조에서는 이점보다 복잡도가 큽니다.

결론:
- 현재 구조에서는 빌드타임 치환이 1순위입니다.

## 3. 월 1회 갱신 절차 초안

1. 월 1회 운영자가 네이버플레이스 공개 후기 중 반영할 문구를 선별합니다.
2. 공개 가능한 표현만 정리해 `data/reviews.json`을 갱신합니다.
3. 이름 익명화, 날짜 형식, 과장 표현 여부를 한 번 더 검수합니다.
4. 다음 배포 사이클에서 빌드타임 치환으로 `#testimonials`를 교체합니다.

운영 메모:
- 신규 후기 급증 시에는 정기 주기 외에 수시 갱신도 가능합니다.
- 후기 원문 복사 여부와 공개 허용 범위는 별도 운영 기준으로 관리하는 편이 안전합니다.
